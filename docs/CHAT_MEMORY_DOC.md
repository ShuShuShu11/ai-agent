# 对话记忆超限处理方案

## 问题背景

`MessageWindowChatMemory` 使用固定窗口策略（maxMessages=20），超过限制后新消息会覆盖旧消息，导致早期上下文丢失。

---

## 最终实施方案：Summary Window + 全量压缩

### 核心思路

对话超过阈值时，将当前**所有消息**压缩为一个摘要，保存摘要 + 近期原始消息。后续触发时，旧摘要会和新消息一起合并再压缩。

### 配置参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `maxRecentMessages` | 20 | 保留的近期原始消息数量 |
| `summarizeThreshold` | 50 | 触发摘要的消息数阈值 |

### 触发流程（全量压缩）

```
┌─────────────────────────────────────────────────────────────┐
│                     消息 1-50                                │
│                 (正常存储，不触发摘要)                         │
└─────────────────────────────────┬───────────────────────────┘
                                  │ 超过50条时触发
                                  ▼
┌─────────────────────────────────────────────────────────────┐
│  第51条消息到达，messages=51 > threshold=50                 │
│                                                             │
│  1. 获取当前所有消息 (51条)                                  │
│  2. 调用 LLM 将 51条消息 全部压缩 → summary                  │
│  3. 保留后 20条 (recentMessages)                             │
│  4. 保存 summary + 20条近期                                  │
│                                                             │
│  结果：1条摘要 + 20条近期 ≈ 21条                              │
│  保留比例：21/51 ≈ 41%                                      │
└─────────────────────────────────────────────────────────────┘
```

### 渐进式压缩示例

```
第1轮（51条时触发）
  - 压缩：消息1-51（全部51条）→ S1
  - 保存：摘要S1 + 消息31-51(20条)
  - 结果：S1 + 20条

第2轮（触发时已有摘要）
  - 压缩：消息1-41 + 旧摘要S1 → S2（合并压缩）
  - 保存：摘要S2 + 消息21-41(20条)
  - 结果：S2 + 20条

第3轮
  - 压缩：消息1-41 + 旧摘要S2 → S3
  - 保存：摘要S3 + 消息21-41(20条)
  - 结果：S3 + 20条
```

---

## 压缩策略对比

| 策略 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **全量压缩** | 每次将所有消息压缩为摘要 | 信息不丢失 ✅ | 成本较高 |
| **增量压缩** | 只压缩被丢弃的旧消息 | 成本低 | 需要维护多个摘要 |
| **分层压缩** | 摘要 + 近期 + 当前会话三层 | 层次清晰 | 实现复杂 |

**本项目采用：全量压缩**

---

## 实现架构

```
FileBasedChatMemory
    │
    ├── 原始消息存储（近期）
    │     └── ./chat-memory/{conversationId}.kryo
    │
    └── 摘要存储
          └── ./chat-memory/{conversationId}.summary.kryo

SummarizingChatMemory（包装 FileBasedChatMemory）
    │
    ├── add(): 添加消息，超过阈值触发全量压缩
    ├── get(): 返回 [摘要] + 近期原始消息
    └── clear(): 清理对话和摘要
```

## 方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **Fixed Window** | 直接丢弃旧消息 | 简单 | 丢失重要上下文 |
| **Sliding Window** | 固定窗口滑动 | 控制成本 | 仍是随机丢弃 |
| **Summary Window** | 旧消息压缩为摘要 | 保留核心信息 | 摘要质量依赖LLM |
| **Token Budget** | 按token预算管理 | 最优成本 | 实现复杂 |

---

## 设计原则

1. **触发阈值 > 保留数量**：避免每次都触发摘要
2. **全量压缩**：每次压缩所有消息，渐进累积
3. **保留比例约 40-50%**：平衡上下文完整性和 token 成本
