# 对话记忆超限处理方案

## 问题背景

`MessageWindowChatMemory` 使用固定窗口策略（maxMessages=20），超过限制后新消息会覆盖旧消息，导致早期上下文丢失。

---

## 最终实施方案：Summary Window + 渐进式压缩

### 核心思路

对话超过阈值时，调用 LLM 将早期消息压缩为摘要，只保留摘要 + 近期原始消息。后续触发时，旧摘要会和新消息一起合并再压缩，实现渐进式累积。

### 配置参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `maxRecentMessages` | 20 | 保留的近期原始消息数量 |
| `summarizeThreshold` | 50 | 触发摘要的消息数阈值 |

### 触发流程

```
┌─────────────────────────────────────────────────────────────┐
│                     消息 1-50                                │
│                 (正常存储，不触发摘要)                         │
└─────────────────────────────────┬───────────────────────────┘
                                  │ 超过50条时触发
                                  ▼
┌─────────────────────────────────────────────────────────────┐
│  第51条消息到达，messages=51 > threshold=50                 │
│                                                             │
│  1. 获取当前所有消息 (51条)                                  │
│  2. oldMessages = subList(0, 31) = 前31条                  │
│  3. recentMessages = subList(31, 51) = 后20条             │
│  4. 压缩 oldMessages → summary                             │
│  5. 保存 summary，清除旧消息，只保留 recentMessages(20条)    │
│                                                             │
│  结果：1条摘要 + 20条近期 ≈ 21条                             │
│  保留比例：21/51 ≈ 41%                                      │
└─────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────┐
│                   消息 52-70                                │
│              (正常添加，消息数 21→39)                        │
│                   触发阈值 50，不触发                         │
└─────────────────────────────────┬───────────────────────────┘
                                  │ 消息71到达，messages=41 > 50? 否！
                                  ▼
                              不触发
```

### 再次触发（消息71-90）

```
┌─────────────────────────────────────────────────────────────┐
│  添加第71条消息后，messages=41                               │
│  41 > 50? → 否，不触发                                       │
│                                                             │
│  ... 添加到第91条时，messages=61 > 50 → 触发                 │
│                                                             │
│  参与压缩的消息：messages - maxRecent = 61 - 20 = 41条      │
│  保留：摘要 + 20条近期 ≈ 21条                                │
└─────────────────────────────────────────────────────────────┘
```

### 渐进式压缩示例

```
第1轮（51条时触发）
  - 压缩：消息1-31
  - 保存：摘要S1 + 消息31-51(20条)
  - 结果：S1 + 20条

第2轮（触发时已有摘要）
  - 压缩：消息1-41（包括之前的摘要S1）
  - 保存：摘要S2 + 消息41-61(20条)
  - 结果：S2 + 20条

第3轮
  - 压缩：消息1-61（包括之前的摘要S2）
  - 保存：摘要S3 + 消息61-81(20条)
  - 结果：S3 + 20条
```

## 实现架构

```
FileBasedChatMemory
    │
    ├── 原始消息存储（近期）
    │     └── ./chat-memory/{conversationId}.kryo
    │
    └── 摘要存储
          └── ./chat-memory/{conversationId}.summary.kryo

SummarizingChatMemory（包装 FileBasedChatMemory）
    │
    ├── add(): 添加消息，超过阈值触发摘要
    ├── get(): 返回 [摘要] + 近期原始消息
    └── clear(): 清理对话和摘要
```

## 方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **Fixed Window** | 直接丢弃旧消息 | 简单 | 丢失重要上下文 |
| **Sliding Window** | 固定窗口滑动 | 控制成本 | 仍是随机丢弃 |
| **Summary Window** | 旧消息压缩为摘要 | 保留核心信息 | 摘要质量依赖LLM |
| **Token Budget** | 按token预算管理 | 最优成本 | 实现复杂 |

---

## 设计原则

1. **触发阈值 > 保留数量**：避免每次都触发摘要
2. **渐进式压缩**：已有摘要时合并压缩，保留历史信息
3. **保留比例约 40-70%**：平衡上下文完整性和 token 成本

### 参考优秀 Agent

| Agent | 保留比例 | 触发阈值 |
|-------|----------|----------|
| Claude Code | ~70% | 1.5-2 倍 |
| Codex / Copilot | ~60-80% | 2 倍 |
| Cursor | ~60% | 2 倍 |
| 本项目 | ~40-60% | 2.5 倍 |
